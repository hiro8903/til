# dbdocs 導入・運用マニュアル

このドキュメントは、ER図（データベースの設計図）をウェブ経由で公開・管理するためのツール `dbdocs` の導入から日々の運用手順までをまとめたものです。

---

## 💻 検証環境 (Environment)

この手順は、以下の環境で動作を確認しています。

- **OS**: macOS Sequoia 15.4.1 (Apple Silicon / M2)
- **Node.js**: v20.11.1 (nodenv管理)
- **Command**: npx (npm 付属)

---

## 🌟 導入するメリット（なぜこれを使うのか？）

データベースの設計を「単なる画像」ではなく「コード（DBML）」として管理することで、以下のメリットが得られます。

1.  **チームへの可視化と共有がスムーズ**
    - [dbdocs.io](https://dbdocs.io/) にデプロイすることで、最新のER図と詳細なテーブル定義をいつでもブラウザで確認でき、URLひとつで共有可能です。
2.  **設計の変更履歴を Git で管理できる**
    - 元データがテキスト（`.dbml` ファイル）なので、画像と違って `git diff` で「どのカラムが増えたか」「リレーションがどう変わったか」を一目で追跡できます。
3.  **「設計を考える」ための専用の場所として機能する**
    - アプリのプログラム（マイグレーションファイル等）と切り離されているため、実際にコードを書く前に「まずは設計案を練る」という思考の整理に集中できます。

> [!CAUTION]
> **運用の注意点：二重管理について**
> このファイルは自動更新されません。**「プログラム（実際のDB構造）」と「ドキュメント（DBML）」を手動で同期し続ける必要があります。** 設計の変更時には、常に両方の更新をセットで行う習慣をつけましょう。

## 🔍 OSS（オープンソース）とサービスについて

初めてこのツールを使う方のために、背景を整理しておきます。

-   **DBML (Database Markup Language)**:
    - オープンソース（OSS）の「言語」です。世界中の誰でも仕様を拡張したり、対応ツールを作ったりできます。
-   **[dbdocs.io](https://dbdocs.io/)**:
    - DBMLを開発した [Holistics社](https://www.holistics.io/) が提供している**無料のホスティングサービス**です。完全にオープンなプログラムというよりは、「便利で安全な公開場所を無料で貸してくれている」という立ち位置です。

## 0. 事前準備：Node.js 環境の確認

`dbdocs` を動かすためには、PCに **Node.js** という実行環境と、それに付随するツールが整っている必要があります。

| 道具名 | 役割 / イメージ | 備考 |
| :--- | :--- | :--- |
| **Node.js** | **「エンジン本体」** | プログラムを動かすのに必須 |
| **npm** | **「道具箱（管理ツール）」** | Node.jsを入れると自動で付いてくる |
| **npx** | **「道具を取り出さずに使う機能」** | npmの中に最初から入っている |

* **まだ環境が整っていない場合**:
  先に [Node.js 環境構築ガイド (setup-nodejs.md)](../[JS]_[Run]_Nodejs/setup-nodejs.md) を読み、環境を構築してください。

* **環境が整っているかの確認方法**:
  ターミナルで以下のコマンドを入力し、バージョン番号が表示されれば準備完了です。
  ```bash
  node -v
  # 例: v20.11.1 と表示されればOK
  ```

## 1. プロジェクト専用ツールのインストール
`dbdocs` を「このプロジェクト専用」として導入します。

* **目的**: PC全体（グローバル）にインストールせず、プロジェクトフォルダ内 (`node_modules`) だけで完結させることで、他のプロジェクトへの干渉を防ぐ「クリーンな管理」を徹底します。

* **コマンド**:
  ```bash
  # 実行ディレクトリ: あなたのプロジェクトフォルダ

  # 1. プロジェクトの初期設定（package.jsonの作成）
  # -y オプションは、すべての質問に Yes で答えるという意味です
  npm init -y

  # 2. dbdocs 本体のインストール
  # -g をつけないことで「ローカル（このフォルダ内）」へのインストールになります
  npm install dbdocs
  ```

* **豆知識**:
  * **package.json**: このプロジェクトで何のツールを使っているかを記した「目録（マニフェスト）」です。
  * **node_modules**: 実際にインストールされたツール本体が格納されるフォルダです。中身は非常に重いため、Git管理からは通常除外されます。

---

## 2. dbdocs へのログイン
設計図をアップロードするためのアカウント連携を行います。

* **コマンド**:
  ```bash
  # npx を使い、プロジェクト内の実行ファイルを直接呼び出す
  npx dbdocs login
  ```
ブラウザが起動するので、GitHub等でログインを完了させてください。ターミナルに「Successfully logged in」と出れば成功です。

> [!NOTE]
> **npx とは？**
> `npm install` でローカルに入れたコマンドを、わざわざパスを指定せずに呼び出せる「実行用の道具」のようなものです。

---

## 3. 設計図（DBML）の作成と初デプロイ
ER図の元となるテキストファイル（DBML形式）を作成し、初めてウェブに公開します。

* **コマンド**:
  ```bash
  # 1. ドキュメント用フォルダの作成
  mkdir -p docs/design

  # 2. 設計図ファイルの作成（中身はVS Code等で編集）
  touch docs/design/sample.dbml

  # 3. 初回のビルド（デプロイ）実行
  npx dbdocs build docs/design/sample.dbml
  ```

### 初回の対話形式
コマンドを打つと、ターミナル上で以下のようにプロジェクト名を聞かれます。

`✔ Project name:` → ここで **任意のプロジェクト名**（例：`visualize-sample-db`）を入力して Enter を押します。

> [!WARNING]
> **名付けの注意点**
> プロジェクト名は **「半角英数字とハイフンのみ」** が推奨されます。スペースや日本語を入れると `Invalid project name!` とエラーが出るので注意してください。

完了すると、専用の公開URL（例: `https://dbdocs.io/yourname/visualize-sample-db`）が発行されます。

> [!TIP]
> **VS Code での編集を快適に**
> DBMLは「Database Markup Language」の略で、非常にシンプルな構文でテーブルを定義できます。VS Codeをお使いの場合は、拡張機能 **「vscode-dbml」** をインストールすると色付け（シンタックスハイライト）が効いて書きやすくなります。

---

## 4. 設計の更新と再デプロイ（運用ルール）
ファイルを修正した内容を、既存のURLに反映させます。

* **手順**:
  1. **ファイルを保存**: VS Code等で `docs/design/sample.dbml` を編集し、保存します。
  2. **再デプロイの実行**:
     ```bash
     # --project オプションで初回に決めた名前を指定するのがポイント
     npx dbdocs build docs/design/sample.dbml --project visualize-sample-db
     ```

* **なぜ --project をつけるのか？**:
  名前を指定せずに実行すると、毎回新しいプロジェクトとして認識されたり、名前を再入力する手間が発生したりします。名前を固定することで、既存のページをスムーズに上書き更新できます。

---

## 5. セキュリティ設定
プロジェクトに閲覧制限（パスワード）を設けます。

* **コマンド**:
  ```bash
  # パスワード設定（閲覧時にパスワード入力を要求します）
  npx dbdocs password --set YOUR_PASSWORD --project visualize-sample-db

  # パスワード解除（誰でも見られる状態に戻します）
  npx dbdocs password --remove --project visualize-sample-db
  ```

## 6. Gitによる構成管理の仕上げ
最後に、この環境構成を GitHub に保存してバックアップを取ります。

> [!IMPORTANT]
> **node_modules を含めない！**
> `node_modules` は Git に含めないのが世界共通の鉄則です。これを含めてしまうと、リポジトリが肥大化し、開発環境が汚れやすくなります。

* **コマンド**:
  ```bash
  # 1. 不要なファイルを Git の対象から外す設定（.gitignoreへの追記）
  # 既にファイルがある場合でも、末尾に付け加える（>>）のが安全です
  echo "node_modules/" >> .gitignore

  # 2. 変更を記録して GitHub へ送信
  git add .
  git commit -m "docs: setup dbdocs environment and project directory structure"
  git push origin main
  ```

### 💡 補足：Git に含めるべきファイル・含めないファイル
*   **含める（コミットする）**:
    *   `package.json` / `package-lock.json`（ツールの目録。これがあれば誰でも `npm install` だけで環境再現できます）
    *   `.node-version`（どのNode.jsを使うかの指示書）
    *   `docs/design/*.dbml`（設計図の本体。最も大事です）
*   **含めない（除外する）**:
    *   `node_modules/`（中身が膨大で、環境ごとに再生成できるため不要です）

## 7. 【応用】新しいRailsプロジェクトへの適用手順
`rails new` で新しいアプリを作成した際、そのプロジェクト内でも `dbdocs` をすぐに活用するためのワークフローです。

* **手順**:
  ```bash
  # 1. 新しいRailsプロジェクト内へ移動
  cd my_new_app

  # 2. Node.js バージョンの固定（指示書を作成する）
  nodenv local 20.11.1
  nodenv rehash

  # 3. プロジェクト専用ツールの導入
  # npm init -y (Railsの場合、既に package.json がある場合は不要です)
  npm install dbdocs

  # 4. Git除外設定の更新（既存の .gitignore の末尾に追記）
  echo "node_modules/" >> .gitignore

  # 5. 設計図の作成とデプロイ
  mkdir -p docs/design
  touch docs/design/schema.dbml
  npx dbdocs build docs/design/schema.dbml
  ```

## 💡 補足：なぜ毎回 npm install するのか？
プロジェクトごとに「道具箱（`node_modules`）」を分けることには3つの大きなメリットがあります。

1. **クリーンな管理**: このプロジェクトをフォルダごと削除すれば、関連するツールもすべて消え、PCにゴミが残りません。
2. **安全な運用**: 他のプロジェクトで行ったアップデートが、このプロジェクトの動作に影響を及ぼす心配がありません。
3. **再現性**: 他の人があなたのプロジェクトをダウンロードした際、`npm install` だけであなたと全く同じツール環境が即座に整います。
