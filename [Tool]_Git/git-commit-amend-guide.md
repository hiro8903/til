# [Tool] Git: 直前のコミットに修正を「重ねる」技術 (amend)

Git で変更を保存（コミット）した直後に、「あ、ファイルを1つ入れ忘れた」「タイポ（誤字）を見つけた」「やっぱりメッセージを書き直したい」といったミスに気づくことは多々ある。
これを一々新しいコミットとして保存するのではなく、**歴史の直前の1ページに綺麗にまとめ直す（上書きする）技術** である `--amend` オプションについてまとめる。

---

## 💡 Context (背景)

TIL のドキュメントをリモートリポジトリに push した後、さらにサンプルのコードを追加し、全体の情報を充実させる修正を行った。この際、別々のコミットとして歴史に残すよりも「1つの完全版の追加」として歴史を綺麗に保つため、`--amend` を使って**過去のコミットに修正内容を重ね合わせた**。このやり分けを使いこなすために記録する。

---

## 🔧 やり方：2つのパターン

前回のコミット（保存）に、いま手元にある修正内容やファイルの追加を合流させたい場合、まず通常の `git add .` を行った後、以下のいずれかのコマンドを打つ。

### パターン 1: コミットメッセージは「そのまま」で、変更だけを重ねる
「誤字を直しただけ」など、過去に保存したメッセージから意味が変わらない場合に使う。

```bash
git add .
git commit --amend --no-edit
```
- **`--no-edit` の意味**: 「メッセージを再編集（edit）しない」という指示。これを付けると、Git は何も聞いてこず、黙って前回のコミットに今の変更を合流させて完了する。

### パターン 2: コミットメッセージも「新しく書き直す」
「追加で大きな修正を入れたため、前回のメッセージでは内容が不足している」という場合に使う。

```bash
git add .
git commit --amend -m "新しい最新のコミットメッセージ"
```
- **`-m` の意味**: この時点で、過去のメッセージは破棄され、ここで指定した「新しいメッセージ」で歴史が上書きされる。

---

## ⚠️ 超重要：上書き後の「Push」のルール

`--amend` は、過去の歴史を「改ざん」する行為である。
もし**「まだインターネット（GitHub 等のリモート）に Push していないコミット」**を amend した場合は、そのまま通常の `git push` で問題ない。

しかし、**「すでにインターネット上に Push されたコミット」**をローカルで手直し・amend した場合は、歴史に矛盾が生じる。インターネット上のGitは「あれ？自分が持っている歴史と違うぞ？」と拒否してくる。

これを解決して、自分の手元の新しい歴史で完全に塗り替えるのが **Force Push（強制上書き）** である。

```bash
git push -f origin main
# （-f は --force の略）
```

### 🚨 チーム開発での注意点
Force Push は文字通りの「強制上書き」であるため、**「他人も触っているブランチ（main 等）」で行うと、仲間の手元にある歴史を破壊してしまう**ためご法度とされている。「自分だけが作業しているブランチ（feature/xxx 等）」や個人のリポジトリでのみ使う、というのがプロの現場の鉄則である。
