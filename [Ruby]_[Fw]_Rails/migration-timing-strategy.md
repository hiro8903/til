# 🏗️ Rails マイグレーションの設計戦略：未定のカラムをどうするか？

🔗 **公式ドキュメント**: [Active Record Migrations - 2.2. Creating a New Table (Rails Guides)](https://edgeguides.rubyonrails.org/active_record_migrations.html#creating-a-new-table)

## 📖 シチュエーション
新規機能（例：認証機能）を構築している最中に、**「ログイン用のIDとパスワードだけ作って、名前(`name`)などは後でユーザー情報編集機能を作るときに別ファイルで追加（`add_column`）すべきか？ それとも今いれるべきか？」**という悩みです。

### 📊 技術的なアドバイス（結論）
基本的には **「今、この瞬間に設計図（ER図）で見えているカラムは、初期のマイグレーションで最初から入れておく」** のが最もスマートな選択です。

*   **無駄な変更（`add_column`）を防ぐ:** 後回しにして細かく `add_column` を繰り返すと、スキーマの履歴が複雑になり、可読性が落ちます。
*   **早い段階で必要になる:** 特に `name`（名前）などは、管理画面を作る際に結局すぐに必要になります（コードだけでは誰かわからないため）。
*   **関連データの制約:** 初期データの作成（`seeds.rb` 等）で必須になる情報も多々あります。

## 🛠️ 具体的なお掃除（整理）の工程

開発中に「やっぱりこれも足しておきたい」と思ったとき、まだ `main` ブランチに合流（プッシュ）していない手元の環境であれば、**自動生成された細切れのファイルを統合して「ひとつの完璧な初期マイグレーション」にまとめ直す**ことができるのも Rails の強みです。

```bash
# 1. ロールバック: いったんDBの状態を「無」の状態（migration直前）に戻す。
# ※ STEP=2 とすれば、直近の2回分を戻すことができる。
$ bin/rails db:rollback STEP=2
```

1.  **古いファイルの削除**: 自動生成された複数の小出しのマイグレーションファイルを直接削除する。
2.  **統合ファイルの作成**: （例）`CreateAuthenticationTables` という名前で、`users` (コード, パスワード, 名前, ロール) を一度に宣言する綺麗なファイルを新しく作成する。
3.  **再マイグレーション**: `$ bin/rails db:migrate` を実行して、DBを最新の「完璧な状態」に更新する。
