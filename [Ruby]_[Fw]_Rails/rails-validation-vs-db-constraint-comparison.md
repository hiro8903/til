# [Ruby] [Fw] Rails: 制約・バリデーション・インデックスの完全マッピング (-comparison)

Rails アプリケーションにおいて、データの正しさを守る仕組みは「モデル（Rails 側）」と「データベース（マイグレーション側）」の 2 箇所に記述できる。実務でよく設定する項目について、それらの役割分担と「なぜそう書くのか」を網羅する。

---

## 🛡 なぜ「二重の守り」が必要なのか？

### 1. ユーザーに親切なエラーを出すため（モデルの役割）
DB 制約だけで守ろうとすると、ユーザーが入力ミスをした際に「システムエラー（500エラー）」のような怖い画面が出てしまう。モデルにバリデーションを書くことで、「名前を入力してください」という具体的なメッセージを画面上に優しく表示できる。

### 2. 万が一の「すり抜け」を防ぐため（DB の役割）
Rails の `validates` は、同時に大量のリクエストが届いた際（レースコンディション）や、開発者が間違えてバリデーションをスキップするコマンドを打った際に、おかしなデータを通してしまうことがある。DB 側に制約があれば、これを物理的に阻止できる。

---

## 🔍 比較表：全体的な役割と特性

| 比較項目 | モデル上のバリデーション (`validates`) | データベース制約 (`null: false`, 等) |
| :--- | :--- | :--- |
| **役割** | **UX（ユーザー体験）の向上** | **データの完全性の担保（最後の砦）** |
| **エラー時** | 保存を中断し、画面にエラーメッセージを出す。 | Rails がシステムエラー(`StatementInvalid`)で停止。 |
| **柔軟性** | 非常に高い（「特定条件のみチェック」等が容易）。 | 低い（テーブル構造そのものを変える必要がある）。 |
| **確実性** | すり抜ける可能性がある（`save(validate: false)` 等）。 | **絶対**。どんな手段を用いても制約違反は許さない。 |

## 🔍 逆引きマッピング：どの制約をどちらに書くべきか？

### 1. 必須チェック（空欄禁止）
- **DB設定**: カラムに `null: false`
- **モデル設定**: `validates :column, presence: true`
- **結論**: **【両方に書く】**
- **理由**: モデルはユーザーに「必須です」と優しく教えるため。DBはプログラムのバグによる空データの混入を物理的に防ぐため。

### 2. 重複禁止（一意性 / Unique）
- **DB設定**: `create_table` ブロックの外側（同じ `def change` 内）に `add_index :table, :column, unique: true` を記述
- **モデル設定**: `validates :column, uniqueness: true`
- **結論**: **【両方に書く】**
- **理由**: 「重複を探す」には全データを調べる必要があるため、DB側に「目次（インデックス）」を作らせる必要がある。そのため、カラム定義ではなく `add_index` を使う。モデルだけだと同時アクセス時にすり抜けてダブることがあるため、DB側のインデックスが必須。

### 3. 親子の連動（親が消えた時の子の挙動）
- **DB設定**: `foreign_key: true`
- **モデル設定**: `has_many :children, dependent: :destroy` (または `:nullify`)
- **結論**: **【両方に書くが、役割は完全に別物】**
- **理由**: 
  - **モデル (`dependent`)**: 子データを1つずつ丁寧に呼び出し、コールバック（画像ファイルの削除処理など）を走らせながら消す「お行儀の良い削除ロジック」。
  - **DB (`foreign_key`)**: どんな手段（直接のSQLなど）を用いても「親がいない子」の存在を許さないための「物理的なブロック（参照整合性の担保）」。

### 4. 検索の高速化
- **DB設定**: `create_table` ブロックの外側に `add_index :table, :column` を記述
- **モデル設定**: なし（モデルではできない）
- **結論**: **【DBにのみ書く】**
- **理由**: `state` や `is_customer`、または外部キー（`user_id`）など、「頻繁に検索（WHERE句）の条件に使われるカラム」には目次を作る。これによりDBの検索スピードが爆発的に上がる。

### 5. デフォルト値の付与
- **DB設定**: カラムに `default: 0` など
- **モデル設定**: `after_initialize` コールバックや `attribute :column, default: 0`
- **結論**: **【基本はDBに書き、動的ならモデルに書く】**
- **理由**: 数値の 0 やフラグの `false` のような固定値は DB 側で設定するのが一番確実で安全。ただし、「今日の日付」や「複雑な計算結果」などを初期値にしたい場合はモデル等で処理する。

### 6. 長さやフォーマット（形式）のチェック
- **DB設定**: 文字列カラムの長さを制限（例: `limit: 20`）
- **モデル設定**: `validates :column, length: { maximum: 20 }`, `format: { with: /.../ }`
- **結論**: **【基本はモデルに書き、重要なものはDBでも制限する】**
- **理由**: メールアドレスの形など複雑なチェックは正規表現が必要なためモデルの得意分野。DBレベルで長さを切ると無理やり収めようとして文字化け（途中切れ）することがあるため、モデルでしっかりバリデーションしてエラーを返すのが親切。

## 🗺️ 視覚でわかる：マイグレーションファイルの構造マップ

「なぜインデックスは別の場所に書くのか？」など、マイグレーションファイル内の**「書く場所と役割」**を以下の図解で把握しておくと迷わなくなる。前章の「逆引きマッピング」の番号と対応している。

```ruby
class CreateExamples < ActiveRecord::Migration[8.1]
  
  # ▼ 【データベースへの変更手順全体】を記述する場所
  def change
    
    # ----------------------------------------------------
    # ①【テーブル（箱）の作成 と カラム（列）の定義場所】
    # ----------------------------------------------------
    create_table :examples do |t|
      # ここには「箱そのものの形や付属ルール」を記述する。
      
      # 1. 必須チェック（空欄禁止）
      t.string :name, null: false
      
      # 3. 親子の連動（外部キー制約）
      t.references :user, null: false, foreign_key: true
      
      # 5. デフォルト値の付与
      t.integer :status, default: 0
      
      # 6. 長さのチェック（DBレベルの制限）
      t.string :short_code, limit: 20
      
      # ❌ NG: unique: true (重複禁止は目次＝インデックスの機能なのでここには書けない)
    end
    # --- テーブル（箱）作り終了 ---

    # ----------------------------------------------------
    # ②【完成したテーブルに対する、追加装備の定義場所】
    # ----------------------------------------------------
    # 箱が完成した「後」に、その箱に対して目次を作ったりする場所。
    
    # 2. 重複禁止（一意性 / Unique）
    add_index :examples, :short_code, unique: true
    
    # 4. 検索の高速化（単なる目次を作る）
    add_index :examples, :status
    
  end # ▼ 変更手順終了
  
end
```

## 💡 まとめ：設計の鉄則

「画面から入ってくるユーザーのミス」は **モデル（Validations）** で優しく受け止め、「プログラムのバグや外部からの不正データ」は **データベース（Constraints / Index）** で鉄壁に守る。
どちらか片方ではなく、重要なルールについては **「まず DB（マイグレーション）で厳しくルールや目次を作り、その上でモデルで優しく案内する」** のがプロの確実な開発手法である。
