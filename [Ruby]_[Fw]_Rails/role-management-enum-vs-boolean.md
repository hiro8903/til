# 🛡️ 権限や状態の管理：Boolean vs Integer (Rails enum)

🔗 **公式ドキュメント一覧**:
*   [Rails API: ActiveRecord::Enum](https://api.rubyonrails.org/classes/ActiveRecord/Enum.html)
*   [Rails Guides: Rails Internationalization (I18n) API](https://guides.rubyonrails.org/i18n.html)

## 📖 概要
システムで「管理者か、一般ユーザーか」といったフラグを持たせるときに、**`boolean`（真偽値: `is_admin` 等）** にすべきか、それとも **`integer`（数値: `role` 等）で定義し `enum` を使うべきか** を比較・検討した記録です。

結論から言うと、**将来少しでも役割（ステータス）が増える可能性があるなら Integer (Enum) の一択** です。

## 🆚 徹底比較

そもそも `enum` とは何か？ という基本については、以下のドキュメントを参照してください。
👉 **[Rails `enum` (列挙型) の基本と魔法のメソッドたち](./enum-basics-and-usage.md)**

---

比較項目 | Boolean (`admin: true/false`) | Integer (`role: 0, 1, 2...`) + `enum` |
| :--- | :--- | :--- |
| **読みやすさ (コード)** | `user.admin?` と直感的に書ける。 | `user.admin?` と同等に直感的（`enum`の恩恵）。 |
| **拡張性 (将来性)** | ❌ **非常に低い**。「閲覧のみ権限」など、3つ以上の役割が必要になると `is_reviewer` などカラムが増え続け、設計が破綻する（アンチパターン）。 | ✅ **非常に高い**。モデルに `reviewer: 2` などを足すだけで簡単にステータスや役割を無限に増やせる。 |
| **可読性 (DB直接操作時)**| `true/false` で人間が見てすぐわかる。 | `0/1` だけだと意味不明になる（コード側での定義書が必要）。 |
| **クエリの書きやすさ** | `where(admin: true)` | `where(role: :admin)` とシンボルで書けて直感的。 |

## 🎨 オススメの実装方法 (Rails `enum` ＋ `i18n`)

製造業のシステムなどでは「工場長」「班長」「検品担当」など役割が細分化されるのが常です。
**「DBは数値で管理し、プログラム上はシンボルで扱い、画面の表示用には『翻訳ファイル (i18n)』を利用する」** というのが最も洗練されたアプローチです。別の「役割マスタテーブル」を作る必要はありません。

詳しい「日本語翻訳ファイル（ja.yml）の書き方」や「_i18n メソッドを使った洗練された実践例」については、独立した以下の TIL を参照して実装を進めてください。
