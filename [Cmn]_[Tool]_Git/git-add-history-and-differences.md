# 🌳 `git add .` vs `git add -A` と「小文字の罠」の歴史的経緯

🔗 **公式ドキュメント一覧**:
*   [Git 公式リファレンス: git-add -A](https://git-scm.com/docs/git-add#Documentation/git-add.txt--A)
    *   ※ `-A` (`--all`) の仕様が記述されています。小文字の `-a` は仕様に存在しません。
    *   ※ **【重要】 `.`（ドット）は専用のオプション項目としては載っていません。** Git にとって `.` はオプションではなく、`<pathspec>`（カレントディレクトリという「対象のパス」）に過ぎないためです。
*   [Git 公式リファレンス: git-commit -a](https://git-scm.com/docs/git-commit#Documentation/git-commit.txt--a)
    *   ※ `git commit -a` (`--all`) が「新規ファイル（untracked）」を無視する仕様について記述されています。

## 📖 概要
Git における変更内容のステージングについて、現在と過去（Git 2.0 以前）でなぜコマンドの推奨が変わったのか、そして意外と引っかかる「小文字・大文字の罠」についてまとめました。

---

## 🛑 1. Gitの「小文字の罠」 (`add -A` と `commit -a`)

コマンドの大文字・小文字は全く意味が異なるため、混同に注意が必要です。公式仕様書の通り、以下の動作となります。

### `git add -A` (大文字 A = All)
*   **意味**: 今居るディレクトリに関わらず、**プロジェクト全体のすべて**の変更（新規・修正・削除）をアド（記録の準備）する。
*   ※ `git add -a` （小文字）というコマンドは標準サポートされておらず、エラーになります。

### `git commit -a` (小文字 a = all)
*   **意味**: `git add` を省略して、いきなりコミットするショートカット。
*   **⚠️ 落とし穴**: 公式ドキュメントにある通り **「新しく作ったファイル（Untracked）」は無視されてコミットされません。** 修正と削除のみが対象です。初心者が「全部入った」と勘違いしやすい罠です。

---

## 🏛️ 2. 歴史的経緯: なぜ昔のエンジニアは `git add -A` を強く推したのか？

現在（Git 2.0以降）は「一番上の階層にいる限り `git add .` でも `git add -A` でも結果は全く同じ」です。しかし、昔はそうではありませんでした。

### 👻 Git 1.x 時代（昔の仕様）
*   **`git add .`** : 新規作成・変更のファイル**だけ**を追加し、**「削除したファイル」は無視**するという恐ろしい仕様でした。（削除したファイルを他の人に同期し忘れる事故が多発）
*   **`git add -A`**: 新規・変更に加え、**削除したファイルも漏れなく追加**しました。
*   このため、昔の教本やシニアエンジニアは「絶対に `-A` を使え！ `.` を使うな！」と口酸っぱく指導していました。

### 🕊️ Git 2.0 以降（現在の仕様）
あまりにも削除漏れの事故が多かったため、Git の仕様が改善されました。
*   **`git add .` も「削除されたファイル」を拾うように進化**しました。
*   これにより、プロジェクトのルート（一番上の階層）で実行する限り、`-A` と `.` は完全に同じ結果（アプリ全体を対象とする）になりました。

---

## 💡 現場のベストプラクティス（現在）

深い階層で作業している時の「立ち位置の違い（ドットは今の階層以下のみ対象、`-A` は上の階層を含む全体が対象）」を除けば、現在は以下がもっとも安全で定番の流れです。

1.  プロジェクトの一番上の階層（ルートディレクトリ）に移動する。
2.  `git add .` を実行する。（新規も削除もすべて拾われる）
3.  `git commit -m "メッセージ"` を実行する。
