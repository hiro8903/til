# Rails 8 アセットビルドの裏側：インストールと設定の仕組み

このドキュメントでは、Tailwind CSS や JS フレームワークを Rails に導入する「インストーラー」が裏側で何を行っているのか、そしてビルドとアセット管理の全体像（メカニズム）を記録します。

---

## 📦 1. インストール手順の「裏側」

公式のインストール手順は魔法のように簡単ですが、トラブルシューティングのためには各コマンドが物理的に何を変更しているかを知る必要があります。

### 🛒 ステップ1：`bundle add`（材料の調達）
- **コマンド**: `bundle add tailwindcss-rails`
- **やってること**: `Gemfile` に依存関係を書き込み、`bundle install` を実行します。
- **イメージ**: 「スーパーで食材（ツール本体）を買ってきて、無造作に冷蔵庫に入れるだけ」の作業です。この段階では Rails アプリ（キッチン）の模様替えは行われず、動作にも変化はありません。

### 🍳 ステップ2：`install` コマンド（キッチンの改造）
- **コマンド**: `bin/rails tailwindcss:install`
- **やってること**: Gem という道具を使って、Rails という家の中に「設定ファイル」という家具を配置し、HTML というお皿にスタイルが載るように配管を通します。

インストーラーを実行せずに手動でやる場合、以下の4つの作業を行うことになります：
1. **設定ファイルの作成**: `app/assets/tailwind/application.css` を作成（Tailwind の設定や `@import` を記述）。
2. **読み込みタグの追加**: `application.html.erb` の `stylesheet_link_tag` を Tailwind が読める形式に変更。
3. **ビルド用ファイルの設定**: `Procfile.dev` を作成（または追記）し、Tailwind のビルド・監視プロセスを定義。
4. **起動スクリプトの作成**: `bin/dev` を作成し、Foreman 経由でサーバーと監視を同時起動するよう設定。

---

## 🏗️ 2. なぜ「ビルド環境」が必要なのか？

HTML タグに直接 `style="..."`（インラインスタイル）を書く場合はブラウザが直接理解できるため、特別な準備はいりません。しかし、Tailwind のような「ユーティリティ指向」の CSS を使う場合は **「ビルド環境」** が不可欠になります。

💡 **料理とコンロで例えると**
> インラインスタイルは「完成した料理」をそのままテーブルに出すようなものです。
> 一方、Tailwind は「注文（クラス名）を受けてから、コンロ（ビルド環境）で必要な分の CSS だけを調理（ビルド）する」仕組みです。コンロが正しく設置・点火されていないと、どれだけ HTML にクラスを書いてもスタイルは出力されません。

## 🎭 3. ビルドを支える「登場人物」の整理

Rails 8 のモダンなフロントエンド環境は、主に2つの重要なプロセス（登場人物）によって成り立っています。

### ① 現場監督：Propshaft（アセット管理）
Rails 8 から標準搭載された、**画像や CSS ファイルを管理・配送する「現場監督」** です。

- **役割**: プロジェクト内のフォルダをパトロールし、「これは配信すべきアセットだ」と認めたものを**台帳（Registry）**に登録します。
- **デバッグ知識**: 監督がファイルを見つけられない（`MissingAssetError` になる）場合、「物理的にファイルが存在するか」だけでなく、「監督の台帳に登録されているか」を確認する必要があります。

### ② 現場リーダー：Foreman（プロセス管理）
開発環境において、複数の仕事を同時に回すための**「現場リーダー」**です。

- **役割**: 「Rails サーバーの起動」と「CSS/JS の監視・常時ビルド」という複数の仕事を、1つのターミナル画面で同時に走らせます。
- **起動フロー**: `bin/dev` を叩く → リーダー（Foreman）が呼ばれる → `Procfile.dev` の指示書を読む → 全員を同時に働かせる。
- **エラー時の注意**: `exec: foreman: not found` と出た場合は、OS レベルのパス問題か、Gemfile に `gem "foreman"` が登録されていない（リーダーが雇われていない）ことが原因です。

## 🔍 4. アセット設定の確認方法（台帳の覗き見）

万が一、ファイルを手動で追加したり構成を変えたりした際に Propshaft の挙動がおかしいと感じたら、**物理パス** と **論理台帳** の両方を確認します。

### A. 監督の「巡回ルート」を確認する
監督がそもそもそのフォルダを探しに行っているか（Load Path）を確認します。
```bash
bin/rails runner 'puts Rails.application.assets.load_path.paths.inspect'
```
※ もしここに `app/assets/builds` 等が含まれていなければ、`config/initializers/assets.rb` 等で明示的に追加する必要があります。

### B. 監督の「公式台帳」を確認する
監督がパトロールを終えて「正式なアセット」として認めたファイルの一覧（Registry）を確認します。
```bash
bin/rails runner 'puts Rails.application.assets.load_path.assets.map(&:logical_path).inspect'
```
※ 物理的にファイル（例：`tailwind.css`）があっても、このリストに出力されない限り、Rails はそのファイルを読み込むことができません。

## 🧟 5. 異常検知：「ゾンビ・ディレクトリ」現象と競合トラブル

過去の事例や特定のツールの不具合によって、**「ファイルは確実にあるのに読み込まれない（MissingAssetError）」** という不可解なトラブルが起きることがあります。その原因の多くが「ディレクトリ名とファイル名の競合」です。

### 🚨 物理的な残骸と「ゾンビ化」
**現象**:
`app/assets/builds/` の中に、ビルドされた `tailwind.css`（ファイル）の隣に `tailwind/`（見知らぬ空のフォルダ）が作成されてしまうケースです。
手動でこのフォルダを削除しても、`bin/dev` を起動するたびに、裏側のツール（監視プロセスや Gem の初期化コード）が良かれと思って**勝手にフォルダを復活させてしまう**現象（＝ゾンビ・ディレクトリ）が発生します。

### 💥 なぜフォルダがあるだけでエラーになるのか？
1. **監督（Propshaft）の習性**: Propshaft はアセットを探索する際、同名のフォルダ（`tailwind/`）を先に見つけると「これは読み込み対象外のフォルダだ」と判定します。
2. **無効化の連鎖**: その直後に正しいファイル（`tailwind.css`）を見つけても、すでに `tailwind` という論理名が「対象外」として処理されてしまっているため、**正規のアセットとしての台帳登録を完全にスキップ（無視）してしまいます。**
3. **結果**: 物理的にファイルが存在しても、監督の「台帳（Registry）」には登録されず、Rails 上からは「存在しないアセット」として扱われてエラーとなります。

### 🛠️ 対処法（クリーンアップの手順）
この「物理と論理の乖離」やゾンビ・ディレクトリに直面した場合、以下の手順で原因を絶ち、残骸を排除します。

1. **プロセスの完全停止（元凶を断つ）**:
   ゾンビを生み出している原因（`bin/dev` や `foreman` などの裏側で動いている監視プロセス）を完全に停止させます。
2. **物理的な競合の排除**:
   エラーの元凶となっている不要な同名フォルダ（競合相手）を削除します。
3. **台帳の再確認（証明）**:
   先述の「B. 監督の公式台帳を確認する」の手順を実行し、対象のファイルが台帳に無事復活したことを確認します。これで確実に読み込まれるようになります。

## ⚖️ まとめ
Tailwind 導入のトラブルシューティングや仕組みの理解においては、
1. **「冷蔵庫（Gem）に食材はあるか？」**
2. **「コンロ（ビルド環境）は点火（bin/dev）されているか？」**
3. **「監督（Propshaft）の台帳に登録されているか？（ゾンビに邪魔されていないか？）」**
の3層に分けて問題を切り分けることが、最も確実なアプローチとなります。
