# Rails 開発における View パーシャル（共通部品）の設計とデータ受け渡し

## 🕯️ 背景 (Background)
Ruby on Rails の View（`.erb`）において、似たようなデザイン（見出し、カード、ボタン等）を各画面でハードコーディング（直書き）すると、デザインの変更や保守が困難になる。これを解消するための「共有パーツ（shared）」の設計と、堅牢なデータの受け渡しパターンを以下にまとめる。

---

## 🏗️ 0. 基本用語の定義：動詞・名詞・住所
View パーシャルを扱う際、混同しやすい 3 つの要素を以下のように整理する。

| 要素 | 役割 | 専門性 | 独立性 |
| :--- | :--- | :--- | :--- |
| **`render`** | **命令（動詞）** | 「描画せよ！」という司令塔。 | JSON 出力や別画面表示にも使える。 |
| **Partial** | **存在（名詞）** | `_` で始まるファイル形式。 | どこのフォルダにあっても良い。 |
| **`shared`** | **住所（宛先）** | 共通部品を置くフォルダ名。 | **名前は任意（自由）**。規約上の慣習。 |

### 💡 「設定より規約 (CoC)」の精神
`shared` というフォルダ名は、システム的に必須ではない。しかし、プロジェクト全体で「共通部品はここ！」と決めておくことで、他のエンジニア（あるいは未来の自分）が迷う時間をゼロにする。これが Rails における **「お作法（Convention）」** である。

---

## 🎯 1. 発注者（View）と職人（Shared）の関係
共通パーツ（Shared Partial(シェアード・パーシャル: 共有部品)）を作る際は、**「役割の分離」** を意識する。

- **職人 (Shared)**: 「どう見えるか（枠組み）」の構築に徹する。中身が何であるかは関知しない。
- **発注業者 (Calling View)**: 職人に注文（`render`）を出し、「何を表示するか」のデータを渡す。

### 実装例: `render` による発注
```erb
<%= render "shared/page_header", 
    title: "ユーザー一覧", 
    subtitle: "全従業員を管理します" %>
```

---

## 🗣️ 2. パラメータの受け渡し：必須と任意
部品側で変数を受け取る際、**「そのデータが絶対に必要か、あるときだけでいいか」** によって書き方を変えるのがプロのお作法である。

| 記述 | 意味 | 届かなかった場合 |
| :--- | :--- | :--- |
| **`<%= title %>`** | **必須**。ないと困るもの。 | **エラー停止** (Undefined variable) |
| **`local_assigns[:subtitle]`** | **任意**。ないかもしれないもの。 | **nil を返す** (安全に処理継続) |

### 💡 なぜ `local_assigns` を使うのか？（Ruby のメカニズム）
エラーを防ぐ要は、Ruby の **「名前（変数）の探し方」と「荷物（ハッシュ）の取り出し方」の違い** にある。

#### 1. 変数 `title` の場合：探索と爆発 (NameError)
- **仕組み**: `<%= title %>` と書くと、Ruby は「`title` という名前の変数（またはメソッド）の中身」を探しに行く。
- **結果**: もし `render` 時に値を渡して **いない** 場合、「そんな名前は知らない！」と **NameError** が発生し、画面全体が止まってしまう。
- **意図**: 「これは必須。渡し忘れたらプログラムを止めてでも教えてほしい」という **厳格な設計**。

#### 2. `local_assigns[:subtitle]` の場合：安全な荷札チェック (nil)
- **仕組み**: `local_assigns` は Rails が用意した「渡された全ての値が入った **ハッシュ（荷物セット）**」である。
- **結果**: Ruby のハッシュは、存在しない鍵（ラベル）を指定してもエラーにならず、そっと **`nil`（空）** を返す性質がある。
- **意図**: 「これは補足情報。あれば表示、なければ無視して進めてほしい」という **柔軟な設計**。

#### 🚨 【注意】 シンボル `:` は「魔法のボタン」ではない
- **間違い**: `<%= :title %>` と書けばエラーにならない？
- **正解**: **ならないが、値も表示されない。** `:` を頭につけると、それは中身ではなく「`title` という **名前そのもの（シンボル）**」という扱いになり、画面には単に文字として `title` と出るだけである。

> [!IMPORTANT]
> **必須なら「変数」、任意なら「local_assigns」**。この使い分けが、バグに強く柔軟な View を作るコツである。

## 💎 5. 深掘り：引数の呼び名と Ruby の魔法

### 正しい用語の使い分け (Terminology)
実務で意思疎通をスムーズにするための、厳密な呼び分けを以下に整理する。

```ruby
render "shared/page_header", title: "詳細"
```

- **キーワード引数**: `title: "詳細"` というセット全体。
- **引数名（キー）**: `title` というラベルのこと。
- **値（バリュー）**: `"詳細"` という中身のこと。
- **ローカル変数**: 部品（Partial）の中で使える状態になった `title` のこと。

### 暗黙のハッシュ (Implicit Hash(インプリシット・ハッシュ))
`render` に複数のデータを渡すとき、なぜ第 3、第 4 の引数として混ざらないのか？
それは Ruby の **「最後の引数が `key: value` 形式なら、自動的に一つのハッシュ（連想配列）にまとめる」** というルールがあるからだ。

- **書いているもの**: `render "path", a: 1, b: 2`
- **解釈されるもの**: `render("path", { a: 1, b: 2 })`

このおかげで、開発者は面倒な `{}` を書かずに、いくつでもデータを「引数」として職人（Shared）に渡すことができる。これを **「Rails（Ruby）の魔法」** と呼び、設定の柔軟性を高める重要な仕組みである。

---

## ⚖️ まとめ
- **名前付け**: `aaa`, `bbb` ではなく `title`, `subtitle` など役割を表す名前をつける（認知負荷の軽減）。
- **共通化の順序**:
    1. 頻出デザイン（PageHeader, Card, Badge）を抽出する。
    2. `shared/` 内にファイルを作成する。
    3. `render` を使って、各画面から呼び出す。

これにより、デザイン変更が必要になった際、一箇所のファイルを直すだけですべての画面に反映される **「真の DRY(ドライ: 同じことを繰り返さない)」** なコードが実現する。
