# 🛤️ [Ruby]_[Fw]_Rails-Migration (マイグレーションの運用と作法)

Rails 開発において、マイグレーションファイルをいかに美しく、かつ管理しやすく運用するかをまとめたガイドです。

---

## 🛤️ 基本の黄金律 (The Golden Rule)

内容を修正・変更する際は、以下の「共有状態」を常に確認します。

| 状況 | 推奨されるアクション | 理由 |
| :--- | :--- | :--- |
| **未共有 (Localのみ)** | `rollback` して **ファイルを直接編集** | マイグレーション履歴を汚れさせず、最初から正解だったように保つため。 |
| **共有済 (Remote/Team)** | **新しいマイグレーションを作成** | 他のメンバーのローカル環境を壊さないため。 |

> **判断基準**: 「このファイルを `db:migrate` した人間が、自分以外に世界に一人でもいるか？」

---

## 🛠️ 実践：マイグレーションの編集と管理

「型を間違えた」「定義を追加したい」といった際に、クリーンに修正する標準的な手順です。

1.  **マイグレーションを戻す**:
    ```bash
    bin/rails db:rollback
    ```
2.  **ファイルを直接編集する**:
    `db/migrate/xxx.rb` を開き、中身を修正します。
    ```ruby
    # 例：references への修正
    t.references :user, null: false, foreign_key: true
    ```
3.  **再実行する**:
    ```bash
    bin/rails db:migrate
    ```

## ✨ 履歴と命名の「美学」

### 1. マイグレーションの結合 (Consolidation)
初期構築時や、密接に関連する一連のテーブル（ドメイン）を一度に実装する場合、1 つのファイルにまとめても構いません。
- **ポイント**: 参照関係がある場合、必ず **「参照される側（親テーブル）」を先に記述** してください。

### 2. 名前に「実態」を反映させる
ファイルを合体させたり役割を拡げた場合は、**「クラス名」と「ファイル名」も変更しておく** のが丁寧な管理方法です。

- **ルール**: 「ファイル名（日付以降）」と「クラス名」は完全に一致させる必要があります。
  - ファイル名: `2026..._create_initial_schema.rb`
  - クラス名: `class CreateInitialSchema`

### 💡 ヒント：大量のテーブルを一気に作る場合
実務では、10枚ものマイグレーションが並ぶと管理が煩雑になるため、**「業務ドメイン（意味のまとまり）」** ごとに 3〜5個 程度に分けるのがスマートです。

- **単位の例**: `CreateAuthenticationTables`, `CreateOrderTables` など。
- **例外**: 初回構築時に限り、あえて `CreateInitialSchema` として一括定義することもあります。

> **コツ**: 1枚にまとめすぎると、1箇所直すだけで全てのテーブルが削除されてしまいます。「修正の単位（やり直しのしやすさ）」を意識して切り分けましょう。

### 💡 コラム：実務での使い分け（コマンドか、手書きか？）
ベテランほど以下のように「ハイブリッド」で使い分けています。

- **器を作る**: カラムは指定せず `bin/rails g model User` とだけ打ち、モデルとマイグレーションの殻を生成。
- **中身を仕上げる**: 生成されたファイルをエディタで開き、落ち着いてカラム定義やインデックス等を「手書き」で丁寧に追記。
