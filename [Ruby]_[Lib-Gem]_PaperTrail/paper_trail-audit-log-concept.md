# 📜 PaperTrail: Rails における「データのタイムマシン（変更履歴管理）」

🔗 **公式ドキュメント**: [GitHub: paper-trail-gem/paper_trail](https://github.com/paper-trail-gem/paper_trail)

## 📖 PaperTrail とは何か？（What / Why）

PaperTrail は、Ruby on Rails アプリケーションでデータベースのレコードが「いつ、誰に、どのように変更されたか」を全て記録する **「変更履歴（監査ログ・Audit Log）」** のデファクトスタンダードとなる Gem です。

### 📌 大きな特徴

1.  **導入が超簡単**: 対象としたいモデル（`User`, `Article`, `Order` 等）にたった 1 行 `has_paper_trail` と書くだけで動作します。
2.  **別テーブルでの一括管理**: 変更履歴は、元のデータのテーブルとは分離された **`versions`** という専用の共通テーブルに自動で溜まっていきます。
3.  **タイムマシン機能**: 間違えてデータを上書き・削除（Destroy）してしまっても、過去の任意の時間（Version）の状態にすぐにデータを戻す（Undo / Restore）ことができます。
4.  **"誰が"（Whodunnit）の追跡**: アプリケーション内の誰がその操作をしたのか、ログイン情報と紐付けて記録できます。

## 🏗️ 一般的な活用事例と仕組み（How / Uses）

PaperTrail がどのようなプロジェクトで必要とされるのか、一般的なシナリオを見てみましょう。

### 1. 監査機能（オーディットログ）としての利用
*   「なぜこの発注数量が10個から100個に変わっているんだ？」というトラブル時に、`version` テーブルを覗くことで **「誰が・いつ・何の項目を・何から何へ変更したか」** という確固たる証拠（監査証跡）を確認できます。
*   金融システム、医療システム、製造業基幹システムなど、**法的・規制的な理由で「データ変更の歴史を絶対に消してはいけない」要件** がある場合に真っ先に採用されます。

### 2. 「元に戻す（Undo）」機能の実装
*   Wiki サイトやブログアプリケーションなど、ユーザーが間違えて記事を消してしまった時に、「リビジョン履歴」として「1つ前のバージョンに戻す」機能を提供するために使われます。

### 3. Versions (履歴) テーブルの中身
PaperTrail が自動で保存するのは、主に以下の情報です。

| カラム名 | 説明 | 例 |
| :--- | :--- | :--- |
| `item_type` | どのモデルの履歴か | `Article` |
| `item_id` | そのモデルの ID はいくつか | `142` |
| `event` | 何の操作をしたか | `create`, `update`, `destroy` |
| `whodunnit` | 誰が実行したか（ID） | `User #5` |
| `object` | 変更 **前** のデータのスナップショット | （YAML / JSON形式） |
| `object_changes`| どの項目が「何から何へ」変わったかの **差分** | `{ title: ["旧題", "新題"] }` |

> ⚠️ **注意点（デメリット）**:
> 履歴データ（`versions`）はテキストの塊（YAMLやJSON形式）として保存されるため、**「2023年に名前が『田中』だった人を全員検索する」といった SQL の検索・集計には全く向いていません。** あくまで「個別のデータの過去を振り返る（復元・監視）」ためのツールです。毎日集計したいデータは、業務用の履歴テーブルを独自に作成すべきです。

## 🏭 製造業アプリ (`mfg_core`) での特殊な活用例

一般的な Web サービスでは「退会したユーザーのデータを復元するため」などに使われますが、私たちの `mfg_core` のような業務システムでは少し異なります。

### 「権限」と「マスタデータ」の監視カメラとして
`mfg_core` には以下の要件があります。
*   一人の社員の配属履歴（いつどの工場にいたか）自体は、検索や集計のために `assignments` という専用の「業務テーブル」で独自に管理している。

**`mfg_core` では PaperTrail をどう使うか？**
「業務の履歴（歴史）」は専用テーブルで綺麗に残しますが、**「その歴史データ自体を、後から誰かがコッソリ改ざんしていないか？」** を監視するために PaperTrail を用います。

1. **システム権限の監視**: 誰かが不正に「システム管理者」の権限を付与（あるいは剥奪）されていないかを見張る。
2. **マスタの監視**: 品番ごとの「標準単価」や「製造レシピ」のような、会社の利益・品質に直結する設計図を**誰が書き換えたか**という証跡を、マスタごとに独自テーブルを作ることなく、`has_paper_trail` を1行書くだけで全マスタに統一して導入する。

これによって、**「毎日の集計が早くて便利な業務システム（専用テーブル）」と「不正やミスを許さない堅牢な監査システム（PaperTrail）」の両立** というプロフェッショナルな設計を実現しています。（※ 開発初期は複雑さを避けるため外しておき、監視が必要になったタイミングで後付けで導入する方針です）
